# Release Evidence Directory

Each current release run must produce:

- `artifacts/releases/<tag>/manifest.json`
- `artifacts/releases/<tag>/checksums.sha256`
- `artifacts/releases/<tag>/rollback/db-rollback-readiness.json`
- `artifacts/releases/<tag>/rollback/drill.json`
- `artifacts/releases/<tag>/canary/canary-rollout-evidence.json`
- `artifacts/releases/<tag>/canary/canary-rollout-dryrun.log`

Validation rules:

1. `db-rollback-readiness.json` must be generated by `scripts/release/verify_db_rollback_readiness.py`.
2. `summary.missing_policy` must be `0`.
3. `summary.blocked_without_down` must be `0`.
4. `drill.json` must include required keys:
   - `release_tag`
   - `executed_at`
   - `executor`
   - `strategy`
   - `result`
   - `migrations_checked`
5. Canary evidence can be generated with:
   - `scripts/release/capture_canary_rollout_evidence.sh [release_tag]`

Canonical evidence rules:

- `artifacts/releases/<tag>/` is an evidence directory, not the release verdict by itself.
- Only the **current run** bundle generated by `release-evidence-attest.yml` is canonical for attestation and release proof.
- Historical sample files under this directory are documentation examples and may be dirty or incomplete.
- Repo-side completion and external/public distribution completion are split; see `docs/reference/done-model.md`.

Tracked historical example rules:

- Any manifest tracked in Git under `artifacts/releases/**/manifest.json` must declare `historical_example=true`.
- Tracked historical manifests must declare `evidence_scope=historical-example`.
- Tracked historical manifests must keep `git.dirty=true` so they cannot be misread as a fresh current-run receipt.
- These tracked files are **historical examples** only.
- They are **not release verdict proof** for the current commit.
