# Testing SLO and Release Readiness

## Scope

This document defines:

- CI KPI fields generated by `scripts/collect_ci_kpi.py`
- Release readiness scoring generated by `scripts/build_release_readiness_report.py`
- Default thresholds for green/yellow/red decisions

## KPI Sources

- JUnit XML artifacts:
  - Example: `.runtime-cache/python-tests-junit.xml`
  - Example: `.runtime-cache/web-e2e-junit-*.xml`
- Coverage artifacts:
  - Python coverage XML (Cobertura style), example: `.runtime-cache/python-coverage.xml`
  - Web summary JSON, example: `apps/web/coverage/coverage-summary.json`
- Mutation stats:
  - Mutmut stats JSON, example: `mutants/mutmut-cicd-stats.json`

## KPI Field Definitions

`collect_ci_kpi.py` writes `ci-kpi-summary.json` and `ci-kpi-summary.md`.

Top-level fields:

- `generated_at`: UTC ISO timestamp
- `inputs.junit_files`: resolved JUnit files list
- `inputs.coverage_xml_files`: resolved Python coverage XML files list
- `inputs.coverage_summary_files`: resolved web coverage summary JSON files list
- `inputs.mutation_files`: resolved mutation stats files list

KPI object fields:

- `kpi.junit.files_total`: matched junit file count
- `kpi.junit.files_parsed`: successfully parsed junit file count
- `kpi.junit.files_broken`: unparsable junit files
- `kpi.junit.tests_total`: total tests
- `kpi.junit.tests_passed`: passed tests
- `kpi.junit.tests_failed`: failure count
- `kpi.junit.tests_errors`: error count
- `kpi.junit.tests_skipped`: skipped count
- `kpi.junit.failed_total`: `tests_failed + tests_errors`
- `kpi.junit.pass_rate_pct`: passed/total ratio in percent
- `kpi.junit.duration_seconds`: summed suite duration

- `kpi.coverage_python.files_total`: matched coverage XML file count
- `kpi.coverage_python.files_parsed`: successfully parsed coverage XML file count
- `kpi.coverage_python.files_broken`: unparsable coverage XML files
- `kpi.coverage_python.lines_covered`: covered lines
- `kpi.coverage_python.lines_valid`: total valid lines
- `kpi.coverage_python.lines_pct`: line coverage percent
- `kpi.coverage_python.branches_covered`: covered branches
- `kpi.coverage_python.branches_valid`: total valid branches
- `kpi.coverage_python.branches_pct`: branch coverage percent

- `kpi.coverage_web.files_total`: matched web coverage summary files
- `kpi.coverage_web.files_parsed`: successfully parsed summary files
- `kpi.coverage_web.files_broken`: unparsable summary files
- `kpi.coverage_web.lines_total`: total lines
- `kpi.coverage_web.lines_covered`: covered lines
- `kpi.coverage_web.lines_pct`: line coverage percent
- `kpi.coverage_web.branches_total`: total branches
- `kpi.coverage_web.branches_covered`: covered branches
- `kpi.coverage_web.branches_pct`: branch coverage percent
- `kpi.coverage_web.functions_total`: total functions
- `kpi.coverage_web.functions_covered`: covered functions
- `kpi.coverage_web.functions_pct`: function coverage percent
- `kpi.coverage_web.statements_total`: total statements
- `kpi.coverage_web.statements_covered`: covered statements
- `kpi.coverage_web.statements_pct`: statement coverage percent

- `kpi.mutation.files_total`: matched mutation stats file count
- `kpi.mutation.files_parsed`: successfully parsed mutation stats file count
- `kpi.mutation.files_broken`: unparsable mutation stats files
- `kpi.mutation.killed`: killed mutants
- `kpi.mutation.survived`: survived mutants
- `kpi.mutation.total`: total mutants from source stats
- `kpi.mutation.effective`: `killed + survived`
- `kpi.mutation.score_pct`: mutation score in percent (`killed/effective`)

## SLO Thresholds (Defaults)

These defaults are used by `build_release_readiness_report.py` and are configurable by CLI flags.

- JUnit pass rate:
  - Green: `>= 100%`
  - Yellow: `>= 98%`
  - Red: `< 98%`
- JUnit failed total:
  - Green: `0`
  - Yellow: `1-2`
  - Red: `> 2`
- Coverage (Python/Web lines):
  - Green: `>= 80%`
  - Yellow: `>= 75%`
  - Red: `< 75%`
- Mutation score:
  - Green: `>= 85%`
  - Yellow: `>= 60%`
  - Red: `< 60%`

## Readiness Scoring Rules

Each check has:

- `status`: `pass|warn|fail`
- `required`: whether failure is blocking
- `weight`: contribution weight

Status-to-score mapping:

- `pass = 1.0`
- `warn = 0.5`
- `fail = 0.0`

Weighted score:

- `score_pct = (sum(weight * status_score) / sum(weight)) * 100`

Release readiness color:

- `red`:
  - any `required=true` check is `fail`, or
  - `score_pct < yellow_threshold` (default 65)
- `yellow`:
  - not red, and (`warn` exists or `score_pct < green_threshold`, default 85)
- `green`:
  - no required fail, no warn, and `score_pct >= green_threshold`

## CLI Usage

Collect KPI:

```bash
python3 scripts/collect_ci_kpi.py \
  --json-out reports/release-readiness/ci-kpi-summary.json \
  --md-out reports/release-readiness/ci-kpi-summary.md
```

Build readiness:

```bash
python3 scripts/build_release_readiness_report.py \
  --kpi-json reports/release-readiness/ci-kpi-summary.json \
  --json-out reports/release-readiness/release-readiness.json \
  --md-out reports/release-readiness/release-readiness.md
```

Optional external checks:

```bash
python3 scripts/release/generate_release_prechecks.py

python3 scripts/build_release_readiness_report.py \
  --kpi-json reports/release-readiness/ci-kpi-summary.json \
  --check-json .runtime-cache/temp/release-readiness/prechecks.json
```
